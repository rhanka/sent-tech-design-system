# Revue adversariale — Lot 3 BI charts (Bullet / Marimekko / ParallelCoordinates / Candlestick / CalendarHeatmap / Bump)

Reviewer: Opus 4.8 (lecture seule, **rien commité/modifié**).
Cibles: `packages/components-svelte/src/lib/{BulletChart,MarimekkoChart,ParallelCoordinatesChart,CandlestickChart,CalendarHeatmapChart,BumpChart}.svelte` + `src/BiCharts3.test.ts`.
Particularité lot 3 : **Svelte uniquement** (pas encore de port React/Vue). Les constats « parité » sont donc *prospectifs* (préparation de l'API canonique pour les futurs ports).

## Résultat des gardes-fous demandés

- **(a) Algos** : corrects en isolé pour Bullet (bandes/cible), Marimekko (largeur∝part / hauteur∝%), Parallel (**normalisation indépendante par axe — vérifiée**), Bump (rank→y inversé, splines C). **DEUX défauts** : mapping date→cellule du CalendarHeatmap (fuseau, #1) et **absence totale de validation OHLC** Candlestick (#2).
- **(b) Tokens inventés** : **PASS**. `grep -E "feedback-danger|surface-muted|spacing-100|spacing-200|spacing-300|data-series"` sur les 6 fichiers = **vide**. Tous les tokens référencés existent dans `themes/css/sent-tech.css` : `--st-semantic-data-category1..8` (Tableau 10), `feedback-success #16a34a`, `feedback-error #dc2626`, `surface-inverse`, `text-inverse`, `border-subtle`, `surface-default`, `text-primary/secondary`. Aucun token fantôme.
- **(c) Contraste labels** : **PASS sur le principe**. Seul Marimekko peint du texte *dans* un fill catégoriel, et il utilise `contrastTextForTone()` (pas de blanc fixe). Calendar n'a aucun texte en cellule ; Bump/Parallel posent les labels *hors* fill. **Réserve mineure** : pour les labels ~10px (texte « normal » → seuil 4.5:1), `contrastTextForTone` choisit toujours la meilleure option mais reste **3,9–4,4:1** sur category1/3/7 (#6).
- **(d) NaN/Infini/vide/négatifs** : **bien gardés**. `Number.isFinite` filtré partout (domaines, sommes, segments) ; Marimekko `Math.abs` sur valeurs négatives ; vide → `[]`/domaine `[0,1]` sans crash. Test « empty calendar » et « no ranges » passent. RAS bloquant.
- **(e) reduced-motion** : **PASS**. Les 6 incluent `@media (prefers-reduced-motion: reduce){ transition: none }` (Bump en a 2 : line + dot). Régression du lot 2 corrigée.
- **(f) a11y** : **PASS**. `role="img"` + `aria-label`, SVG `aria-hidden="true"` + `focusable="false"`, `ChartDataList` (clip-rect SR-only) présent dans les 6. Réserves : infobulles `role="presentation"` (cohérent avec ChartDataList comme source SR) ; Bump expose les rangs manquants `#?` en SR (#3).
- **(g) Edge cases** : 1 seul point, valeurs égales, dates non triées (Calendar **trie**), 1 seule catégorie (Bump) → testés, sans crash. **Défauts** : rangs manquants Bump (#3), labels dupliqués (clé `#each`, #4), domaine dégénéré Candlestick (#7).
- **(h) API width/height** : **cohérente** (`width`/`height` numériques partout, pas de `size` unique). Bonne base pour parité future. Réserves d'alignement futur en #8.

## Suite de tests

`npx vitest run src/BiCharts3.test.ts` → **15/15 PASS** (env. `America/Toronto`, UTC−5). **MAIS** ces tests ne couvrent ni le placement weekday du Calendar, ni la cohérence OHLC, ni les libellés dupliqués, ni les rangs manquants. Les verts sont donc trompeurs sur #1–#4 (voir détail #1 : le bug fuseau **ne casse pas le compte de cellules** mais décale la ligne jour-semaine).

## Top 8

| # | Composant | Constat | Sévérité | Correction |
|---|-----------|---------|----------|------------|
| 1 | **CalendarHeatmap** | **Mapping date→cellule incohérent (mélange local/UTC).** La ligne jour-semaine vient de `cur.getDay()` (**heure locale**), mais la clé de valeur et le `data-chart-date`/tooltip viennent de `cur.toISOString().slice(0,10)` (**UTC**). En fuseau à offset **positif** (Europe/Paris, Asia/Tokyo, Sydney — la majeure partie du globe), `new Date("2024-01-01T00:00:00")` = `2023-12-31T23:00:00Z` : **vérifié** que la valeur du 1er janvier (un **lundi**, `getDay()=1`) est rangée dans la cellule dont `toISOString="2024-01-01"` qui tombe sur la **ligne mardi (`getDay()=2`)**. Résultat : **toutes les valeurs glissent d'une ligne jour-semaine** et le libellé de date de l'infobulle est décalé. Le test ne vérifiant que *le nombre* de cellules non vides, il passe en CI (souvent UTC) **et** trompeusement à Paris alors que le rendu est faux. | **Élevé** | Tout dériver d'un calendrier **cohérent** : soit tout en UTC (`Date.UTC`, `getUTCDay`), soit tout en local (construire `dateStr` via `getFullYear/getMonth/getDate` locaux, pas `toISOString`). Ne jamais mélanger `getDay()` local et `toISOString()` UTC. |
| 2 | **Candlestick** | **Aucune validation de cohérence OHLC.** Le composant fait confiance aux entrées : si `high < max(open,close)` (ou `low > min(open,close)`), **vérifié** que le corps déborde hors de la mèche (ex. open100/close110/high105/low95 ⇒ haut du corps `y=78.7` *au-dessus* de la mèche haute `y=112` → bougie visuellement aberrante). Le domaine n'est calculé que sur `high`/`low` (l.86-88) : si ceux-ci sont incohérents avec open/close, le corps sort carrément du plot. Aucune alerte, aucun clamp. | **Élevé** | Normaliser par bougie : `hi=max(o,h,l,c)`, `lo=min(o,h,l,c)` pour le domaine et la mèche, ou garde-fou explicite `high≥max(open,close)≥min(open,close)≥low` (clamp + signalement). |
| 3 | **Bump** | **Rang manquant rendu silencieusement comme #1.** `rankToY(undefined)` ⇒ `Math.max(1, undefined→NaN-guarded)` retombe sur `1` ⇒ **vérifié** : le point est dessiné en **haut (rang 1)**. Or la liste SR affiche `#?` (l.122). **Désync visuel/SR** : un trou de données apparaît comme « 1ʳᵉ place ». Idem si `ranks` est plus court que `categories` (les `ci` au-delà donnent `undefined`). | **Moyen** | Pour un rang non fini : **ne pas tracer** le point/segment (rompre la polyligne) plutôt que le forcer à #1. Aligner le visuel sur le `#?` du SR. |
| 4 | **Bump / Candlestick / Bullet / Marimekko** | **Collisions de clé `#each` sur labels dupliqués.** Bump : `{#each series as s (s.label)}` et `{#each categories as cat (cat)}`. Candlestick/Bullet : `(c.datum.label)` / `(b.datum.label)`. Deux séries (ou deux périodes/catégories) de même libellé ⇒ Svelte lève *"keyed each duplicate key"* (crash en dev) ou fusionne des nœuds. Les données BI réelles ont fréquemment des libellés répétés (deux "Autre", deux "T1"). | **Moyen** | Clé d'index stable : `(s.index)` / `(i)` / `(`${'`'}…-${'`'}i`)`. (Marimekko clé déjà par `cell.key` composite — OK, mais `{#each data as datum (datum.label)}` l.191 reste vulnérable aux catégories dupliquées.) |
| 5 | **Marimekko** | **La dimension *largeur* (part de catégorie) est absente du SR.** L'élément `ChartDataList` ne rapporte que `"EMEA, Product A: 60%"` — le **%** du segment *dans sa colonne*, jamais la part de largeur de la colonne ni la valeur absolue. Or l'intérêt d'un Marimekko est l'**aire 2D** (largeur × hauteur). Un lecteur d'écran perçoit donc un simple stacked-100% et perd l'information que porte le graphe. | **Moyen** | Inclure la part de colonne dans le SR : p.ex. `"EMEA (40% du total), Product A : 60% du segment"`. |
| 6 | **Tous fills catégoriels (Marimekko)** | **Contraste limite pour les labels ~10px.** `contrastTextForTone` choisit toujours la **meilleure** option (vérifié : il prend le côté à plus haut ratio sur les 8 tons), mais pour du texte « normal » (<18px, seuil 4.5:1) les ratios restent sous le seuil sur **category1 (4,35), category3 (3,94), category7 (4,24)** — conformes seulement au seuil « gros texte » (3:1). Non bloquant (labels courts, %), mais imparfait. | **Faible** | Soit n'afficher le label en cellule qu'au-delà d'une taille garantissant 3:1 (déjà un seuil `w>28 && h>14`), soit grossir/épaissir la police des labels de cellule, soit assombrir `DARK_TEXT`. |
| 7 | **Candlestick / Bullet** | **Domaine dégénéré (toutes valeurs égales).** Si `high===low` partout (ou une seule bougie plate), `niceTicks(min===max)` renvoie `[base]` ⇒ `domainMin===domainMax` ⇒ `scaleLinear` renvoie `r0` ⇒ toutes bougies/barres s'écrasent en haut du plot (`bodyH=0.5`). Pas de crash, mais rendu illisible et non centré. Idem Bullet si toutes les valeurs/cibles/ranges valent 0 (`domainMax=1` sauve le cas nul, mais pas le cas « tout = 5 »). | **Faible** | Sur domaine plat, élargir artificiellement (`±5%` ou `[v−1, v+1]`) pour centrer verticalement la série. |
| 8 | **Tous (préparation parité)** | **API canonique à figer avant les ports React/Vue.** Points à verrouiller pour éviter les divergences du lot 2 : Bump utilise `data: {label,ranks,tone}` + `categories` séparé (cohérent), Parallel `axes:{key,label,min?,max?}` + `data: Record` + `tone: string[]` (sérié par ligne), Bullet `ranges?: number[]` (interprétés comme **bornes cumulées triées**, pas largeurs). Ces conventions non triviales (sens de `ranges`, `tone` indexé par ligne, rank 1 = haut) doivent être documentées/testées pour que React/Vue s'alignent. Perf : `axisDomain` du Parallel est recalculé **par (ligne×axe)** → O(lignes²·axes) (négligeable en BI, à mémoïser si gros volumes). | **Faible/Info** | Geler ces contrats dans la doc API + ajouter des tests de sens (`ranges` triées, `tone[i]` par ligne) qui serviront de référence aux ports. Mémoïser `axisDomain` par axe. |

## Verdict

**PUBLIABLE SOUS RÉSERVE** — nettement meilleur que les lots 1–2 : zéro token inventé, zéro NaN qui fuit, `reduced-motion` partout, a11y complète, contraste géré par `contrastTextForTone` (pas de blanc fixe), normalisation par axe du Parallel **correcte**. **Aucun bloquant.** Mais **deux défauts Élevés** doivent être corrigés avant publication : le **mélange local/UTC du CalendarHeatmap** (#1) range les valeurs sur la mauvaise ligne et fausse les dates d'infobulle dès qu'on quitte un fuseau ≤ UTC — défaut invisible des tests actuels et donc piégeux ; et **l'absence de validation OHLC du Candlestick** (#2) produit des bougies aberrantes sur entrées incohérentes. Corriger #1 et #2, puis idéalement #3–#5 (qualité données réelles), avant de figer l'API (#8) pour les ports React/Vue. #6–#7 peuvent suivre.

## Top 5 (priorité de correction)

1. **CalendarHeatmap — fuseau (#1)** : tout dériver d'un seul référentiel temporel (UTC *ou* local cohérent) ; ne jamais croiser `getDay()` local et `toISOString()` UTC. Ajouter un test sous `TZ=Europe/Paris` qui vérifie *la ligne jour-semaine et la date d'infobulle*, pas seulement le compte de cellules.
2. **Candlestick — validation OHLC (#2)** : domaine et mèche sur `max/min(o,h,l,c)` ; garde-fou `high≥max(open,close)≥min(open,close)≥low`.
3. **Bump — rang manquant (#3)** : rompre la ligne au lieu de placer le point en #1 ; aligner le visuel sur le `#?` du SR.
4. **Clés `#each` (#4)** : passer aux clés d'index (Bump séries+catégories, Candlestick, Bullet, Marimekko catégories) pour survivre aux libellés dupliqués.
5. **Marimekko SR (#5)** : injecter la part de largeur (part de catégorie) dans `ChartDataList` pour restituer l'aire 2D aux lecteurs d'écran.
